编码约定与护栏

在 pure-admin 基线上的 MFM 定制约定与 Review 门禁。

路径与别名

  • @/ → src/
  • 业务 API：src/api/admin/
  • 业务视图：src/views/business/
  • 动态路由 component 字符串相对于 src/views/（不含 .vue 后缀）

相对 pure-admin 的 MFM 改动

文件 改动要点
src/api/user.ts 登录/refresh 指向 /admin/auth/*
src/api/routes.ts GET /admin/menus
src/api/admin/collections.ts 集合 CRUD REST
src/utils/http/index.ts VITE_SERVER_BASEURL、Admin 鉴权白名单
src/views/business/collections/ 通用 Mongo 文档管理页
src/views/login/index.vue 开发环境虚拟登录入口

HTTP 分层

views/*.vue → api/admin/*.ts → utils/http → mfm-api
  • 页面/组件不直接 import axios
  • 类型优先放在 API 模块旁（如 CollectionMeta），避免全局 any
  • 错误提示由 http 拦截器统一处理；页面层仅处理业务流程（如 JSON 解析失败）

权限

  • 路由级：meta.roles，后端菜单与 pure-admin filterNoPermissionTree 配合
  • 按钮级：v-perms / RePerms，数据来自登录返回的 permissions
  • 服务端仍以 Admin JWT + 超管/邮箱白名单为最终门禁

命名

  • Vue 组件 defineOptions({ name: "BusinessCollections" }) 与路由 name 一致（keep-alive）
  • API 函数：动词 + 资源，如 fetchRecordsPage、saveRecord
  • 环境变量：VITE_* 前缀，密钥勿提交仓库

禁止模式

禁止 原因
第二套 HTTP 客户端 鉴权/refresh 逻辑分叉
调用 C 端 API 写业务库 绕过 Admin 白名单与审计
前端硬编码可写集合列表 与后端 env 漂移
生产启用 vite-plugin-fake-server mock 与真实 API 不一致

  • pnpm typecheck — tsc + vue-tsc
